# 告警生成与处理流程

## 1. 业务目标

当检测结果或设备状态满足风险条件时，平台要能够及时生成告警、通知相关人员，并为后续处理与审计保留记录。

## 2. 告警来源

- AI 检测结果触发
- 设备状态异常触发
- 业务规则或阈值触发

当前平台文档首版主要关注 AI 检测告警链路。

## 3. 主流程

1. Detection Server 在推理过程中识别到目标类别、置信度和连续出现条件满足告警阈值。
2. Detection Server 根据本地告警触发逻辑做频率控制、去重或级别判断。
3. Detection Server 生成告警事件，并附带会话、时间、类别、截图或其他上下文。
4. Detection Server 将告警事件回调给 DISys。
5. DISys 负责将告警持久化，并与任务、设备、工作空间或人员信息关联。
6. 前端从 DISys 或实时通道中读取告警数据并展示到告警面板或相关页面。
7. 人员查看告警后进行确认、处置或归档。

## 4. 参与系统职责

### Detection Server

- 判定是否触发告警
- 控制告警频率与去重
- 组织截图、检测类别、会话信息等上下文

### DISys

- 保存告警业务记录
- 关联设备、任务、工作空间
- 承担后续查询、审计和通知扩展能力

### DroneCloudSystem-web

- 实时展示和查询告警
- 提供告警查看、筛选和处置入口

## 5. 异常与风险点

- 告警规则前后端口径不一致，会导致展示结果与实际触发逻辑不一致
- Detection Server 回调失败，会导致前端已看到检测结果但后端无持久化记录
- 截图上传失败，会导致告警上下文不完整
- 如果规则仅存在浏览器本地，可能出现跨设备不可见的问题

## 6. 当前已知约束

- 告警链路横跨三个仓库，任何字段变化都需要联动检查
- 部分告警规则能力在前端存在本地化技术债，后续应评估服务端统一管理
- 告警是否触发、如何去重、如何升级需要持续和业务口径对齐
